# GPT 登录凭证详解：AT、RT、Session 与 JSON 文件的生命周期

>  适用范围：ChatGPT 网页端、Codex CLI/IDE，以及使用 JSON 导入账号的第三方客户端。

在账号导入、格式转换和故障排查中，经常会看到 `AT`、`RT`、`Session`、`auth.json` 等名称。它们并不是同一种东西：有的是访问凭证，有的是刷新凭证，有的是浏览器登录状态，还有的只是保存这些数据的文件格式。

本文沿用站内常见的“GPT 登录凭证”叫法。严格来说，本文讨论的是 ChatGPT 与 Codex 客户端的身份验证；GPT 模型本身没有独立的账号登录。除特别说明外，下文中的 AT 指交互式登录流程中的 **OAuth Access Token**，不指面向自动化场景单独创建的 **Codex Access Token**。

::: danger 凭证等同账号权限
Access Token、Refresh Token、认证 Cookie、会话凭证、API Key 和包含这些字段的 JSON 文件都可能代表账号访问权限，应按密码保管。示例中统一使用占位符；不要把真实值上传到网盘、在线解析网站、聊天群、工单或代码仓库。
:::

## 先看结论

- **AT（OAuth Access Token）**：客户端访问受保护服务时使用的凭证，通常有效期较短。
- **RT（Refresh Token）**：AT 失效或过期后，用来申请新 AT 的凭证；并非每种登录流程都会提供 RT。
- **Session**：服务端和浏览器共同维持的登录状态，通常依赖 Cookie 或 Session Token，并不等同于 OAuth 的 RT。
- **API Key**：OpenAI Platform 的调用凭证，按 API 账户计费，不等同于 ChatGPT 订阅登录。
- **Codex Access Token**：面向受信任自动化任务单独创建的 Codex 工作区凭证，不等同于 OAuth 登录缓存中的短期 AT。
- **ID Token**：用于向客户端说明“当前登录者是谁”的身份断言，通常不作为业务 API 的访问凭证。
- **JSON 文件**：保存凭证与账号元数据的容器。文件能被导入，只代表格式被识别，不代表其中的凭证仍然有效。

最常见的误解是：把“有 JSON 文件”理解成“账号可以永久使用”。真正决定能否持续使用的是凭证是否有效、客户端是否支持刷新、RT 是否真实可用，以及账号是否拥有目标功能的权限。

## 一、先分清六个层次

登录体系可以拆成六层。排错时先判断问题发生在哪一层，比反复转换 JSON 更有效。

| 层次 | 解决的问题 | 常见内容 |
| --- | --- | --- |
| 身份验证 | 证明“你是谁” | 密码、邮箱验证码、MFA、SSO、浏览器授权 |
| 身份断言 | 向客户端描述已登录用户 | ID Token 及其 claims |
| 登录状态 | 让网页记住已经登录 | Session、认证 Cookie、Session Token |
| 访问授权 | 允许客户端访问特定服务 | OAuth AT、scope、audience |
| 凭证续期 | 在授权仍有效时申请新 AT | RT、客户端绑定、客户端认证 |
| 本地存储 | 将凭证保存并交给软件读取 | `auth.json`、导入 JSON、系统钥匙串 |

这六层彼此关联，但作用不同。混用这些层次会产生以下误判：

- 网页仍然保持登录，不代表另一个客户端里的 AT 仍然有效。
- JSON 导入成功，不代表 RT 能够刷新 AT。
- AT 有效，不代表账号拥有目标模型、套餐或工作区权限。
- ChatGPT 订阅有效，不代表 OpenAI Platform API Key 可以免计费调用。

## 二、AT：真正用于访问服务的凭证

AT 是 Access Token 的简称。按照 OAuth 2.0 的定义，客户端将 AT 提交给资源服务器，以访问授权范围内的资源。

很多 AT 属于 Bearer Token：谁持有它，谁就可能在有效期和权限范围内使用它。因此，泄露 AT 的影响不只是“别人知道了一个字符串”，而是别人可能直接继承这段凭证代表的访问能力。

### AT 的典型特点

- 通常比 RT 的有效期短，但具体时长由服务端决定。
- 可能是 JWT，也可能是不透明字符串，仅凭外观不足以判断。
- 即使 JWT 中存在 `exp` 字段，也只代表令牌声明的到期时间。
- 服务端可以在到期前撤销令牌，也可以拒绝权限范围不匹配的令牌。
- 客户端通常通过 `Authorization: Bearer <TOKEN>` 一类方式发送 AT。

`Bearer` 描述的是“持有凭证即可使用”的认证方式，不负责说明凭证来自哪里。OpenAI API Key 也通过 Bearer 请求头发送，因此仅凭 `Authorization: Bearer ...` 不足以判断其中一定是 OAuth AT。

Bearer Token 应通过 TLS 传输，并优先放在 `Authorization` 请求头中。不要把 token 放入 URL 查询参数，否则它可能进入浏览器历史、代理日志和服务端访问日志。

::: tip 解码不等于验证
JWT 的载荷可以在本地解码，但解码只负责读取声明。签名是否有效、服务端当前是否接受这个令牌，仍需分别验证。真实令牌应始终留在受信任的本地环境中。
:::

### AT 失效后的常见表现

- 客户端突然要求重新登录。
- 请求返回 `401` 或“token expired”“invalid token”等错误。
- JSON 仍可导入，但账号一启动就失败。
- 软件短时间可用，过一段时间后不再恢复。

最后一种情况说明客户端后续认证流程失败。缺少可用 RT 是常见方向之一，也可能是凭证被撤销、权限变化、网络异常，或客户端没有正确刷新和保存新凭证。

## 三、RT：用于续期，但不是永久通行证

RT 是 Refresh Token 的简称。它只应提交给授权服务器，用于申请新的 AT，而不应直接发送给普通资源服务器。

一个简化的生命周期如下：

```text
用户完成登录
    |
    v
授权服务器签发 AT，并可能同时签发 RT
    |
    v
客户端携带 AT 访问服务
    |
    +---- AT 仍有效 ----> 继续使用
    |
    +---- AT 过期 ------> 客户端提交 RT，并按要求证明客户端身份
                              |
                              +---- 刷新成功 --> 保存新 AT，继续使用
                              |
                              +---- 刷新失败 --> 需要重新登录
```

### 为什么有 RT 仍可能刷新失败

- RT 已过期、被撤销或与当前客户端不匹配。
- RT 绑定于获发它的客户端，导入到另一个客户端后可能不满足绑定或认证要求。
- 客户端只导入了字段，却没有实现正确的刷新流程。
- 登录流程没有签发真实 RT，文件中只是空字符串或占位值。
- 账号、工作区或权限状态已经变化。
- 服务端启用了 RT 轮换，旧 RT 在刷新后立即失效。
- 多个软件复制并同时使用同一份 RT，其中一个完成轮换后，其他副本仍在使用旧值。

OAuth 允许授权服务器在每次刷新时签发新 RT，并撤销旧 RT。因此，“把同一份 JSON 导入很多设备”不一定能提高可用性，反而可能造成凭证相互覆盖或轮流失效。

::: warning RT 通常比 AT 更敏感
AT 泄露后的影响往往受剩余有效期限制；真实 RT 可能继续换取新 AT。保存、传输和备份 RT 时，应使用与密码相同或更严格的保护措施。
:::

## 四、Session：网页登录状态，不等于 RT

Session 不是单一、统一的文件格式。它通常表示服务端认可的一段登录状态，浏览器通过 Cookie 或 Session Token 在后续请求中证明自己属于这段会话。

实际交流中，“Session”至少可能指三种东西：

1. **浏览器会话**：浏览器 Cookie 与服务端状态共同维持的网页登录。
2. **Session Token**：放在 Cookie 或其他存储中的会话标识符。
3. **Session JSON**：某个网页内部接口返回的当前用户、账号、过期时间和 token 快照。

第三种只是某个系统在某个时间点输出的 JSON 对象，并不是 OAuth 标准规定的通用交换格式。字段名称、返回内容和可用范围都可能变化。

### 为什么网页能用，导入客户端却失败

浏览器 Session 和客户端 OAuth 缓存可能是两套状态：

- 浏览器能够继续发送有效 Cookie，所以网页仍然登录。
- 第三方客户端只拿到了短期 AT，没有真实 RT。
- AT 的 audience、scope 或账号权限不适合目标客户端。
- 目标软件要求通过自己的 OAuth 登录流程完成设备或账号验证。

因此，“网页在线”只证明当前浏览器 Session 仍被接受；其他客户端能否复用这段状态，需要按各自的认证流程验证。

## 五、ID Token、API Key 和账号 ID 分别是什么

### ID Token

ID Token 来自 OpenID Connect，主要用于向客户端描述已登录用户的身份及认证信息。它常见于 OAuth 登录缓存，但一般不是访问业务接口时使用的 AT。

部分转换工具会在缺少 ID Token 时生成占位结构，让目标软件能够解析 JSON。这样的占位值只满足格式要求，账号权限和刷新能力仍以真实凭证及服务端响应为准。

### API Key

API Key 属于 OpenAI Platform API 的认证方式，与 ChatGPT 网页订阅是两套使用和计费路径。

API Key 由 Platform 侧创建、限制、轮换和撤销，不存在“使用 RT 自动刷新 API Key”的 OAuth 生命周期。虽然 API Key 和 OAuth AT 都可能放在 Bearer 请求头中，但二者的签发方式、权限来源和计费路径不同。

以 Codex 为例，官方支持两类常用登录方式：

- 使用 ChatGPT 登录，使用对应工作区和订阅权限。
- 使用 OpenAI Platform API Key，按照 API 账户用量计费。

因此，ChatGPT Plus/Pro 等订阅与 API Key 的余额、计费方式需要分别确认。

### Codex Access Token

Codex 官方还使用“Access Token”指代另一类凭证：ChatGPT Business 和 Enterprise 工作区成员可以为受信任的本地自动化创建 Codex Access Token，并通过 `CODEX_ACCESS_TOKEN` 或 `codex login --with-access-token` 使用。

它与本文前文所说的 OAuth AT 不同：

- OAuth AT 通常由交互式登录和刷新流程在客户端内部管理。
- Codex Access Token 由允许的工作区成员主动创建，面向脚本、调度任务和私有 CI。
- Codex Access Token 应设置有效期、定期轮换，并只放在受信任的运行环境中。
- 一般 OpenAI API 调用仍应使用 Platform API Key，而不是把 Codex Access Token 当成通用 API Key。

### Account ID

`account_id`、`chatgpt_account_id`、`workspace_id` 一类字段通常只是标识账号或工作区。它们帮助客户端选择正确的上下文，完整登录仍需对应的认证凭证。

## 六、JSON 文件只是容器

JSON 负责组织数据，不负责让凭证变得有效。不同工具可以把同一类凭证保存成完全不同的字段结构。

### OAuth 标准 token 响应

OAuth 授权服务器的 token 响应可能包含以下字段：

```json
{
  "access_token": "<ACCESS_TOKEN>",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "<REFRESH_TOKEN>",
  "scope": "<SCOPE>"
}
```

其中 `access_token` 和 `token_type` 是标准响应字段，`refresh_token` 是否出现由授权服务器决定。示例中的 `3600` 只用于说明 `expires_in` 以秒为单位，不代表 GPT 或 Codex 的固定有效期。

### 网页 Session JSON 示例

下面的结构来自第三方项目 GPTSession2CPAandSub2API 的输入示例，核对版本为提交 [`a097eb1`](https://github.com/gtxx3600/GPTSession2CPAandSub2API/commit/a097eb155bb7bdf6cbbc26f1e4e75e120ab3163c)。它不代表稳定的 OpenAI 官方接口：

```json
{
  "user": {
    "email": "user@example.com"
  },
  "expires": "2026-08-01T12:00:00.000Z",
  "account": {
    "id": "<ACCOUNT_ID>",
    "planType": "<PLAN>"
  },
  "accessToken": "<ACCESS_TOKEN>",
  "sessionToken": "<SESSION_TOKEN>"
}
```

这类文件可能有 AT 和 Session Token，但不一定包含 OAuth 客户端需要的 RT。当第三方客户端只消费导出的静态 AT，且不接入原网页 Session 或其他刷新流程时，其使用周期通常截止到该 AT 失效。

### Codex 本地凭证缓存

Codex 官方文档说明，CLI 登录凭证可以保存在 `CODEX_HOME` 下的 `auth.json`，也可以保存在操作系统钥匙串中。`CODEX_HOME` 默认是 `~/.codex`。

常见位置：

```text
macOS/Linux: ~/.codex/auth.json
Windows:     %USERPROFILE%\.codex\auth.json
```

Codex 会在实际使用过程中、令牌到期前自动刷新 ChatGPT 登录令牌。若使用文件存储，`auth.json` 中包含访问凭证，应当按密码保护。官方文档没有将其内部字段结构声明为稳定的公开接口，因此不应依赖固定字段手动拼装或修改该文件。

### 第三方导入 JSON

第三方工具会定义自己的导入结构。以 GPTSession2CPAandSub2API 提交 `a097eb1` 支持的输入为例，同一类字段可能分别写成：

```text
accessToken
access_token
tokens.access_token
credentials.access_token
```

格式转换通常只做以下工作：

1. 找到源文件中的 token 和账号元数据。
2. 按目标工具要求重命名或重组字段。
3. 如果 token 是可解析的 JWT，并且包含所需 claims，则尝试读取过期时间或账号元数据。
4. 对缺失字段留空，或写入目标工具能够解析的占位值。

转换过程不会延长 AT 的寿命，也不会把 Session Token 变成真实 RT，更不会增加账号原本没有的权限。

## 七、如何判断一份 JSON 能否持续使用

| 文件内容 | 可能的使用结果 | 主要限制 |
| --- | --- | --- |
| 只有 AT | 可能立即可用 | AT 过期后通常需要重新登录或补充刷新流程 |
| AT + 真实 RT | 客户端可能自动续期 | 取决于客户端刷新实现、权限和服务端策略 |
| AT + 空 RT | 表面字段完整 | 刷新请求通常会失败 |
| AT + 占位 RT | 可能通过格式校验 | 服务端通常会拒绝占位值 |
| Session Token/Cookie | 可能维持特定网页会话 | 不等同于通用客户端 OAuth 凭证 |
| API Key | 可用于支持该 Key 的 API 客户端 | 独立计费，不继承 ChatGPT 订阅额度 |
| 只有账号 ID、邮箱、套餐字段 | 只能提供元数据 | 不具备认证能力 |

判断时不要只看“字段是否齐全”，还要回答四个问题：

1. 当前 AT 是否仍被服务端接受？
2. RT 是否真实存在，并且仍然有效？
3. 目标客户端是否支持这套登录和刷新流程？
4. 账号是否拥有目标模型、套餐和工作区权限？

## 八、过期时间怎么看

不同格式可能使用不同字段：

| 字段 | 常见含义 | 注意事项 |
| --- | --- | --- |
| `exp` | JWT 中的 Unix 秒级过期时间 | 只存在于部分 JWT，解码不代表验证成功 |
| `expires_in` | token 响应生成后还可使用的秒数 | 是相对时长，不是固定日期，也不是所有格式都会保存 |
| `expires` | ISO 8601 格式的过期时间 | 可能代表网页 Session，而不是 AT |
| `expires_at` | 工具计算或保存的到期时间 | 确认单位是秒、毫秒还是日期字符串 |
| `expired` | 工具生成的展示或状态字段 | 不一定由服务端实时返回 |
| `last_refresh` | 最近一次刷新时间 | 不是下一次过期时间 |

本地时间只能用于初步判断。即使显示“尚未过期”，凭证仍可能因为撤销、权限变更、账号状态变化或客户端不匹配而失败。

## 九、从错误现象定位问题

下表是常见判断方向，不同客户端的错误文案可能不同。

| 现象 | 优先检查 | 常见原因 |
| --- | --- | --- |
| JSON 导入失败 | 文件格式层 | JSON 语法错误、字段名不匹配、外层数组或对象结构错误 |
| 导入成功但立即要求登录 | 访问授权层 | AT 无效、凭证类型不被目标客户端接受、账号上下文不匹配 |
| 可以使用一段时间，之后永久失败 | 刷新流程 | 缺少 RT、RT 无效、客户端未保存刷新后的新凭证 |
| 多个客户端先后掉线 | RT 轮换 | 服务端采用 RT 轮换并废弃旧值时，多个副本可能继续使用旧 RT |
| 网页正常，Codex 显示未登录 | 登录状态层 | 浏览器 Session 与 Codex 本地缓存相互独立 |
| 返回 `401` | 认证凭证 | AT 过期、格式错误、签名或 audience 不匹配 |
| 返回 `403` | 权限与策略 | 凭证可能有效，但账号、工作区或功能权限不足 |
| 返回 `429` | 额度与速率 | 请求过多或额度受限，不一定是登录失效 |
| 聊天记录不可见 | 账号上下文或本地数据 | 登录了不同账号/工作区，或会话索引与本地数据发生变化 |

对于官方 Codex CLI，可先运行：

```bash
codex login status
```

需要清除当前本地登录时使用：

```bash
codex logout
```

浏览器登录、Codex 登录和 API Key 应分别检查，不要通过反复覆盖 JSON 来替代定位。



## 十、常见问题

### 有 RT 就一定不会掉登录吗？

不一定。RT 可能被撤销、轮换或拒绝，客户端也可能没有正确实现刷新和持久化。

### Web Session 能直接转换成 Codex 长期凭证吗？

格式转换本身不足以得出这个结论。网页 Session JSON 可能只有 AT 和 Session Token，没有真实 RT。转换后的文件即使能被解析，也可能在 AT 过期后停止工作。

### `auth.json` 可以直接复制到另一台电脑吗？

从文件格式上可能可行，但会扩大凭证暴露面，也可能受到客户端版本、RT 轮换、系统存储方式和账号验证策略影响。更稳定的方式是在目标设备上完成正常登录。

### 为什么套餐正常，却返回无权限或额度不足？

登录成功只解决身份与认证问题。模型权限、工作区、套餐额度、速率限制和 API 计费属于其他层次，需要分别确认。
